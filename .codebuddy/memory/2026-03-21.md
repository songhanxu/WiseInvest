### 自选列表"一闪而过"然后变空的竞态条件修复
**现象**：在搜索页添加自选后关闭搜索框，自选列表中一闪而过刚添加的项目，然后又变成空列表
**根因**：竞态条件（Race Condition）
1. `addToWatchlist` 发起 POST 请求（异步），同时乐观更新了 `watchlist`
2. 关闭搜索框 → `.onChange(of: showSearch)` 立刻触发 `loadWatchlist()` 发起 GET 请求
3. GET 请求先于 POST 完成 → 后端数据库中还没有新记录 → 返回空列表
4. 空列表覆盖乐观更新 → "一闪而过"现象

**iOS 端修复**（`MarketDetailViewModel.swift`）：
- 新增 `pendingMutations` 计数器，追踪正在进行中的 add/remove 请求数量
- `addToWatchlist` 和 `removeFromWatchlist` 调用时 `pendingMutations += 1`，回调完成时 `-= 1`
- `loadWatchlist()` 检测到 `pendingMutations > 0` 时，延迟 1 秒再发起 GET 请求，确保 POST 已持久化
- 拆分为 `loadWatchlist()`（带延迟保护）和 `forceLoadWatchlist()`（直接加载）

**后端修复**（`stock_handler.go` `GetWatchlist`）：
- 在返回 `stocks` 前增加 `nil` → `[]StockResponse{}` 保护，确保 JSON 永远返回 `[]` 而非 `null`

**数据流闭环保障**：
- 序列号机制：**过期请求 → 丢弃**（不写入 @Published 属性 → 不触发视图更新 → 无闪烁）
- 去重机制：**前一个还在飞 → 跳过**（不会堆积请求 → 不会加重后端压力）
- 静默刷新：自动刷新**不显示骨架屏**（不修改 isLoading → UI 无变化 → 数据原地更新无闪烁）

### 竞态策略升级：精确匹配 → 高水位（High Watermark）
**改动原因**：原先 `capturedSeq == currentSeq` 只接受"最后一个请求"的结果，如果发了 5、6、7 三个请求，6 先回来也会被丢弃（因为 currentSeq 已经是 7）。改为高水位机制后，只要比当前已采纳的数据更新就接受。

**新机制**：
- 每个数据源维护两个值：`counter`（递增分配）+ `highWaterMark`（已采纳的最高序号）
- 发请求前：`counter += 1`，捕获 `capturedSeq = counter`
- 回调中：`guard capturedSeq > highWaterMark`，大于则采纳并提升水位，否则丢弃
- **移除** `isRefreshingX` 去重标志 — 允许多个请求同时在飞，谁新谁生效

**涉及文件**：
- `MarketDetailViewModel.swift`：`indicesSeq` → `indicesCounter` + `indicesHighWaterMark`，`watchlistSeq` → `watchlistCounter` + `watchlistHighWaterMark`
- `StockDetailView.swift`：`quoteSeq` + `isRefreshingQuote` → `quoteCounter` + `quoteHighWaterMark`

**示例场景**：当前展示的是 seq=4 的数据，发出了 5、6、7 三个请求：
- 6 先回来 → 6 > 4 ✅ 采纳，水位提升到 6
- 5 后回来 → 5 > 6 ❌ 丢弃（比已展示的旧）
- 7 最后回来 → 7 > 6 ✅ 采纳，水位提升到 7

### 行情更新频率优化（后端缓存 + 前端提频）
**问题**：数据更新速度太慢，无法跟上交易所实时行情节奏
**根因**：前端每 3-5 秒轮询 → 后端每次实时调第三方 API（腾讯/东方财富/币安） → 第三方 API 延迟 + 网络延迟 → 实际数据延迟可达 5-10 秒

**后端改动**：
1. **新增** `infrastructure/cache/quote_cache.go`：Redis 行情缓存封装（SetJSON/GetJSON），支持任意 struct 序列化
2. **`StockHandler` 重构**：
   - 注入 `*cache.QuoteCache`（通过 `NewStockHandler` 第三个参数）
   - 新增 `backgroundRefreshLoop`：启动 3 个后台 goroutine，分别以 1s/2s/2s 间隔轮询 A 股/美股/加密货币指数数据，写入 Redis（TTL 5-10s）
   - `GetIndices` 接口：优先从 Redis 读缓存（毫秒级），缓存未命中才实时调第三方
   - `GetStockQuote` 接口：同样优先读缓存（TTL 2s），未命中时实时获取并写回缓存
   - 新增 `Stop()` 方法，优雅关闭后台 goroutine
3. **`main.go`**：创建 `QuoteCache` 注入 `StockHandler`，`defer stockHandler.Stop()`

**前端改动**：
1. **`MarketTradingHours.swift`**：刷新间隔大幅缩短
   - A 股：3s → **1s**
   - 美股：5s → **2s**
   - 加密货币：5s → **2s**
2. **`StockDataService.swift`**：HTTP 超时缩短
   - 请求超时：8s → **3s**
   - 资源超时：15s → **6s**

**优化效果**：
- 后端：第三方 API 调用从"每个客户端请求都调"变为"后台 1-2 秒调一次，所有客户端读缓存"，API 响应从 ~500ms 降到 ~5ms
- 前端：轮询间隔从 3-5 秒缩短到 1-2 秒，配合缓存毫秒级响应，数据延迟从 5-10 秒降至 1-3 秒
- 减少第三方 API 调用量：N 个客户端共享同一份缓存，不再 N 倍放大请求
